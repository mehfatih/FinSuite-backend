# Data Corruption / Wrong KPI Values

**Symptom:** A merchant reports that an insight, KPI, or daily brief shows numbers that don't match their books. E.g. "MRR is showing 12K but I have 18K in invoices."

**This is sensitive.** Customers lose trust in AI features faster than in any other surface. Treat as P1.

**Affected surfaces:**
- D-1 Insight cards / `Insight` rows generated by aiBriefController
- D-5 morning brief
- D-6 weekly report narrative numbers
- D-8 chat tool replies (`/zyrix mrr`, `/zyrix runway`)

**Confirm — diagnostic queries:**
1. Pull the merchant's id from the support ticket.
2. **MRR sanity check:**
   ```sql
   SELECT SUM(amount) FROM invoices
   WHERE "merchantId" = '<id>'
     AND status IN ('SENT','PAID')
     AND "issueDate" >= date_trunc('month', now());
   ```
   Compare against the value in the Insight body / chat reply.
3. **Cash balance:**
   ```sql
   SELECT SUM(amount) FROM bank_transactions WHERE "merchantId" = '<id>';
   ```
4. **Recent insights:**
   ```sql
   SELECT id, type, title, "generatedAt" FROM insights
   WHERE "merchantId" = '<id>' ORDER BY "generatedAt" DESC LIMIT 10;
   ```

**Likely root causes (most → least common):**
1. **Currency mismatch.** Merchant has both TRY and USD invoices; KPI assumed single-currency. Check `Merchant.currency` vs invoice currencies.
2. **Date window mismatch.** "MRR" in `KPI_COMPUTATIONS` is current month; merchant is asking about trailing 30 days. Reconcile the definitions in the chat reply or brief.
3. **Stale cache.** D-5 brief was generated yesterday using yesterday's data; merchant is comparing to live data. Force a re-gen via the admin morning-brief panel.
4. **Status filter mismatch.** Some KPIs include only PAID; some include SENT+PAID. Document which.

**Fix order:**
1. **If the data is truly wrong** (computation has a bug): DO NOT edit `kpiComputations.ts` autonomously — it's a protected file. Open a Linear ticket; tag Mehmet; he decides the fix.
2. **If the data is correct but the merchant is reading it differently:** explain in their support reply; consider adding a tooltip to the KPI card defining the window/currency.
3. **If a single merchant's data is genuinely corrupt** (e.g. duplicate invoice rows from a bad import): write a one-off SQL fix in a new migration file `prisma/manual-migrations/YYYY-MM-DD_fix_merchant_<id>.sql` and apply via Railway's SQL console.

**Communicate:**
- Customer reply within 4 hours during business hours, even if just "we're investigating; preliminary root cause within 24h."
- Do NOT promise refunds before the root cause is known.

**Resolve:** customer confirms the displayed numbers match their expectation.
